Part F: Dynamic Page Renderer
Context:
MuseKit's page builder allows admins to create unlimited pages (About, Features, Pricing, Contact, etc.) from the admin dashboard, each built from the same 13-section toolkit. These pages are stored in the site_pages table with their sections config in a JSONB column. The CMS package needs components to render these pages on the frontend and provide dynamic navigation.

Required:
1. DynamicPage component (src/landing/DynamicPage.tsx)
Props: slug: string, preview?: boolean
Behavior:
Query site_pages via Supabase for the row where slug matches and status='published'
Preview mode: If preview=true, bypass the status='published' filter (fetch the page regardless of status). Preview mode is used by admins to preview draft/review pages. The integration Repl's route handler will verify the user is an authenticated admin before passing preview=true.
If found: render <LandingPageBuilder sections={page.sections} />
If not found: render a 404 message (e.g., "Page not found") or return null so the app's not-found page handles it
While loading: show a loading skeleton or spinner
If in preview mode and page is not published, show a subtle banner at the top: "Preview Mode — This page is not published" (dismissible)
Export from src/landing/index.ts and src/index.ts
2. SiteNav component (src/landing/SiteNav.tsx)
Props: className?: string (for styling flexibility)
Behavior:
Query site_pages where show_in_nav=true AND status='published', ordered by sort_order
Render navigation links: /<slug> for each page, EXCEPT the homepage (slug="home") which links to /
Handle loading state gracefully — show nothing or a minimal skeleton, never flash an empty nav
Cache the query result (the nav doesn't change often — use React state, no need to re-query on every render)
Export from src/landing/index.ts and src/index.ts
3. HomePageLoader component (src/landing/HomePageLoader.tsx)
Props: none
Behavior:
Query site_pages where slug='home' and status='published'
If found: render <LandingPageBuilder sections={page.sections} />
If not found: fall back to <LandingPageBuilder sections={defaultLandingConfig} /> (use the existing static config as fallback so the homepage always renders something)
While loading: show a loading skeleton
Purpose: This replaces the static getLandingConfig() call in the homepage. The integration Repl will use this component instead of directly calling getLandingConfig().
Export from src/landing/index.ts and src/index.ts
Part G: SEO Architecture (CRITICAL — This is why Next.js was chosen over Vite)
Every public-facing page must be server-rendered with complete SEO metadata. This is the primary architectural reason the project uses Next.js instead of Vite/React. Google and social media crawlers must see full metadata without executing JavaScript. All SEO functions are exported from the CMS package so the integration Repl can wire them into Next.js generateMetadata and route handlers.

G1. Server-Side Metadata Generation
Export async functions that return Next.js Metadata objects. These run server-side before the page renders, ensuring crawlers see complete metadata in the HTML response.

import type { Metadata } from "next";
// For dynamic pages (About, Features, Contact, etc.)
export async function getPageMetadata(slug: string): Promise<Metadata>
// For the homepage specifically
export async function getHomePageMetadata(): Promise<Metadata>
// For individual blog posts
export async function getBlogPostMetadata(slug: string): Promise<Metadata>
// For the blog list page
export async function getBlogListMetadata(): Promise<Metadata>
// For legal pages (terms-of-service, privacy-policy)
export async function getLegalPageMetadata(slug: string): Promise<Metadata>
The getLegalPageMetadata function:

Maps the slug to a human-readable title (e.g., "terms-of-service" → "Terms of Service")
Returns title: "Terms of Service | {appName}" (query settings for branding.appName)
Returns description: A short summary appropriate for the legal page type
Returns robots: { index: true, follow: true } (legal pages should be indexed)
Returns alternates.canonical: {baseUrl}/{slug}
No OG image needed — use branding logo as fallback via branding.logoUrl
Each function must:

Query Supabase server-side (use createServerClient or createAdminClient — NOT the browser client)
Return a complete Metadata object with:
title — from seo_title (if set) or title, appended with site name: "Features | MuseKit". Query settings table for branding.appName.
description — from seo_description (if set) or description
openGraph — { title, description, images: [og_image or branding logo fallback], siteName, type: 'website' } (use 'article' for blog posts)
twitter — { card: 'summary_large_image', title, description, images }
alternates.canonical — from canonical_url (if set) or auto-generated: https://{domain}/{slug}
robots — { index: !no_index, follow: !no_index }
icons — { icon: faviconUrl, apple: faviconUrl } where faviconUrl comes from branding.faviconUrl setting. If no favicon is set, omit the field (Next.js will fall back to /favicon.ico automatically).
Query settings table for branding fallbacks: branding.appName, branding.logoUrl, branding.faviconUrl, branding.description — use these when page-specific values are missing
G2. Dynamic Sitemap
Export an async function for generating a complete sitemap:

export async function generateSitemap(baseUrl: string): Promise<Array<{
  url: string;
  lastModified: Date;
  changeFrequency: 'always' | 'hourly' | 'daily' | 'weekly' | 'monthly' | 'yearly' | 'never';
  priority: number;
}>>
This function:

Queries all site_pages where status='published' AND no_index=false
Queries all posts where published=true
Includes static routes: /pricing, /login, /signup
Includes legal pages: /terms-of-service, /privacy-policy — priority 0.2, changeFrequency 'yearly'
Returns entries with priorities: homepage=1.0, other pages=0.8, pricing=0.7, blog posts=0.6, auth pages=0.3, legal pages=0.2
Uses updated_at as lastModified
The integration Repl will use this in apps/web/src/app/sitemap.ts (Next.js built-in sitemap route)

G6. Heading Hierarchy
The LandingPageBuilder must enforce proper HTML heading structure for SEO:

Only ONE <h1> per page — typically the hero section headline
Section titles (Feature Cards title, FAQ title, Process Steps title, etc.) must use <h2>
Sub-items within sections (individual feature card titles, FAQ questions, step titles) must use <h3>
Update any section components that violate this hierarchy
The hero section should receive a prop or context indicating whether it's the first section (gets <h1>) or a subsequent hero (gets <h2>)
Blog posts: When rendering Markdown blog post content, ensure the post title renders as <h1> and any Markdown headings (## Heading) render as <h2> and below. If the Markdown content contains # H1 headings, downshift them to <h2> to avoid duplicate H1 tags on the page (the post title is already the H1). This can be handled by the Markdown renderer configuration (e.g., remark plugin or custom heading component that maps h1 → h2, h2 → h3, etc.).
